# Data Model: Project Setup Foundation

## Overview

This feature models the development foundation rather than customer-facing
business data. The core entities describe the repository layout, service
boundaries, configuration contracts, local infrastructure profile, validation
entry points, and deployment targets that together make the Keystone MVP
repeatable for contributors.

## Entities

### Repository Workspace

| Field | Type | Rules |
|---|---|---|
| `workspace_id` | String | Stable repository identifier |
| `name` | String | Required; `keystone` for the initial repository |
| `default_branch_pattern` | String | Must support feature branches generated by Speckit |
| `local_first` | Boolean | Required; true for this foundation |
| `aws_hosting_target` | Boolean | Required; true because AWS is the planned hosted target |

Validation rules:
- Exactly one repository workspace exists for the MVP foundation.
- The workspace must support both local development and a later AWS deployment
  target without changing its top-level structure.

### Service Area

| Field | Type | Rules |
|---|---|---|
| `service_id` | String | Primary identifier |
| `name` | Enum | `frontend`, `backend`, `worker`, `shared`, `deploy` |
| `runtime_type` | Enum | `node`, `python`, `shared_assets`, `ops_support` |
| `owner_scope` | String | Required descriptive ownership boundary |
| `can_run_independently` | Boolean | Required |
| `depends_on_local_infra` | Boolean | Required |

Validation rules:
- `frontend`, `backend`, and `worker` must each support an independent
  development loop.
- `shared` and `deploy` exist to prevent cross-service duplication rather than to
  become runtime services themselves.

### Shared Configuration Profile

| Field | Type | Rules |
|---|---|---|
| `profile_id` | String | Primary identifier |
| `environment_name` | Enum | `local`, later `aws` |
| `source_file` | String | Required committed example file |
| `secret_values_committed` | Boolean | Must remain false |
| `service_overrides_allowed` | Boolean | Required |
| `last_validated_at` | Timestamp | Nullable until validation exists |

Validation rules:
- Local setup must derive from committed examples, not handwritten secret files.
- Secrets and machine-specific values must remain outside version control.

### Local Dependency

| Field | Type | Rules |
|---|---|---|
| `dependency_id` | String | Primary identifier |
| `capability` | Enum | `postgresql`, `s3_compatible_storage`, `sqs_compatible_queue`, `scheduler_support` |
| `provider_name` | String | Required; `PostgreSQL`, `MiniStack`, `ElasticMQ`, or local process name |
| `runtime_mode` | Enum | `docker_container`, `local_process` |
| `endpoint_config_key` | String | Required for networked dependencies |
| `health_check` | String | Required command or endpoint expectation |
| `required_for_full_startup` | Boolean | Required |
| `aws_equivalent` | String | Required migration target |

Validation rules:
- Every required local dependency must have a documented health check and a
  matching environment contract.
- MiniStack and ElasticMQ may be separate local providers, but both must preserve
  direct AWS mappings for storage and queue behavior.

### Developer Command

| Field | Type | Rules |
|---|---|---|
| `command_id` | String | Primary identifier |
| `name` | String | Required root or service command name |
| `scope` | Enum | `root`, `frontend`, `backend`, `worker`, `infra` |
| `purpose` | Enum | `bootstrap`, `start`, `lint`, `format`, `test`, `health`, `cleanup`, `provision` |
| `requires_infra` | Boolean | Required |
| `expected_outcome` | String | Required |

Validation rules:
- Root commands must exist for bootstrap, infrastructure lifecycle, formatting,
  linting, testing, provisioning, and health checks.
- Service-level commands must remain available for focused development.

### Validation Suite

| Field | Type | Rules |
|---|---|---|
| `suite_id` | String | Primary identifier |
| `runtime_scope` | Enum | `frontend`, `backend`, `worker`, `root` |
| `suite_type` | Enum | `lint`, `format`, `unit`, `integration`, `smoke` |
| `blocking_for_review` | Boolean | Required |
| `execution_entrypoint` | String | Required |

Validation rules:
- Each runtime must have at least one blocking validation path before review.
- Root smoke validation must confirm service-to-dependency connectivity.

### Deployment Target

| Field | Type | Rules |
|---|---|---|
| `target_id` | String | Primary identifier |
| `name` | Enum | `local`, `aws` |
| `compute_mode` | Enum | `process`, `container_task` |
| `storage_mode` | Enum | `postgres`, `s3_compatible` |
| `queue_mode` | Enum | `sqs_compatible` |
| `orchestration_mode` | Enum | `lightweight_local`, `aws_native_scheduled_tasks` |

Validation rules:
- `local` is mandatory for MVP development and must be fully documented.
- `aws` is the planned successor target and must preserve the same service
  responsibilities rather than redefine them.

## Relationships

- One `Repository Workspace` contains multiple `Service Area` records.
- One `Repository Workspace` exposes one or more `Shared Configuration Profile`
  records.
- A `Service Area` can depend on multiple `Local Dependency` records.
- A `Developer Command` may target one `Service Area` or the whole `Repository
  Workspace`.
- A `Validation Suite` belongs to one `Service Area` or the root workspace.
- `Local Dependency` and `Deployment Target` together define the local-to-AWS
  migration seam.

## State Transitions

### Shared Configuration Profile

`draft_example` → `documented` → `validated`

- Profiles begin as committed examples.
- They become `documented` once setup guidance references them.
- They become `validated` once health and startup flows confirm they work.

### Local Dependency

`defined` → `bootable` → `provisioned` → `healthy` → `stopped`

- A dependency is `defined` when its local contract exists.
- It becomes `bootable` when Docker Compose or equivalent runtime commands exist.
- It becomes `provisioned` once buckets, queues, or other required local assets
  have been created.
- It becomes `healthy` when the documented health check passes.
- It returns to `stopped` when the local environment is shut down cleanly.

### Validation Suite

`configured` → `runnable` → `passing` or `failing`

- Validation starts as `configured` when tooling is committed.
- It becomes `runnable` once contributors have an entry point.
- A run ends in either `passing` or `failing`, which informs review readiness.
